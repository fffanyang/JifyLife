# JifyLife UI 设计方案

> **设计哲学：一个窗口，说出来就好**
>
> 不要让用户思考"这条该记在哪"——JifyLife 只有一个输入口，语音或文字，说完即走。AI 在幕后完成分类、整理、提醒。

---

## 1. 设计原则

JifyLife 的界面设计遵循三个核心原则，一切交互决策都以此为锚：

**第一原则：零决策输入。** 用户打开应用时，不需要选择"创建日程"还是"写备忘"还是"记流水账"。只有一个输入框，说话或打字，内容自动被 AI 理解和归类。这是 P 人最需要的——不要让我选，让我直接说。

**第二原则：语音优先。** 手机端的主要输入方式是语音。点击麦克风按钮，说完自动识别、自动处理。文字输入作为备选。这符合"随手记"的产品定位——走路时、做饭时、躺在床上时，说一句话就完成记录。

**第三原则：渐进式展示。** 主界面极简，只有输入框和时间线。更多内容（日记、打卡统计、知识库）通过轻量的手势或点击"浮现"出来，而不是作为固定的导航目的地。

---

## 2. 整体布局：单窗口架构

整个应用只有 **一个主窗口**，所有功能通过该窗口的不同"层"来呈现。没有底部 Tab 导航，没有侧边栏菜单，没有多级页面跳转。

```
┌─────────────────────────────────────┐
│  JifyLife              [头像] [···] │  ← 顶栏：品牌 + 用户 + 设置
│─────────────────────────────────────│
│                                     │
│  ┌─ 时间线卡片流 ───────────────┐   │
│  │                               │  │
│  │  14:30  "下午三点开会"        │  │  ← AI 解析为日程
│  │         → 已创建日程 15:00    │  │
│  │                               │  │
│  │  14:15  "买了杯美式，真好喝"  │  │  ← AI 标记为流水账
│  │         ☕ 咖啡 · 流水账       │  │
│  │                               │  │
│  │  13:00  "记一下：护照下周二到 │  │  ← AI 解析为备忘 + 提醒
│  │          期要去续签"          │  │
│  │         → 备忘 + 提醒 3/17    │  │
│  │                               │  │
│  │  ...更早的记录...             │  │
│  │                               │  │
│  └───────────────────────────────┘  │
│                                     │
│  ┌───────────────────────────────┐  │
│  │  说点什么...          🎤  ➤  │  │  ← 底部输入栏：文字 + 语音
│  └───────────────────────────────┘  │
│                                     │
└─────────────────────────────────────┘
```

整个界面由三部分构成，从上到下分别是**顶栏**、**时间线卡片流**和**底部输入栏**。时间线占据绝大部分屏幕空间，是用户浏览自己记录的主视图。底部输入栏始终固定可见，是唯一的输入入口。

## 3. 核心交互区域

### 3.1 底部输入栏（The One Input）

这是整个应用最重要的组件。它的设计需要同时满足文字输入和语音输入两种模式。

**默认状态：** 一个圆角输入框，左侧 placeholder 文字"说点什么..."，右侧并排两个图标按钮——🎤（麦克风）和 ➤（发送）。输入框获得焦点时弹出键盘。

**语音输入状态：** 点击麦克风后，输入框区域变为语音录制界面。显示波形动画和实时转写的文字。用户说完话后，短暂停顿自动结束录制（也可手动点击停止）。转写结果填入输入框，用户可以编辑后发送，也可以直接发送。

**发送后的处理流程：**

1. 内容立刻出现在时间线顶部，显示为"处理中..."状态
2. 后端 AI 分析内容意图（日程/备忘/流水账/知识/打卡），耗时约 1-3 秒
3. 卡片更新为最终状态，显示 AI 的解析结果和归类标签
4. 如果 AI 不确定，显示轻提示让用户确认或修改分类

```
输入流程状态机：

  [空闲] ──点击输入框──→ [文字输入模式]
    │                         │
    │                     输入文字 + 点击发送
    │                         │
    ├──点击麦克风──→ [语音录制模式]    │
    │                    │            │
    │              说话 + 停顿/点停止  │
    │                    │            │
    │                    ▼            │
    │              [转写预览模式] ──确认──→ [发送处理]
    │                    │                    │
    │                  编辑修改                │
    │                    │                    ▼
    │                    ▼              [AI 分析归类]
    │              [文字输入模式]              │
    │                                         ▼
    └─────────────────────────────── [卡片展示在时间线]
```

### 3.2 时间线卡片流（Timeline）

时间线是所有记录的展示区域，按时间倒序排列（最新在最上）。每条记录都是一张卡片，卡片的样式根据 AI 归类的类型略有不同：

| 卡片类型 | 视觉标识 | 核心信息 | 可交互动作 |
|---------|---------|---------|-----------|
| **日程** | 左侧蓝色竖线 + 📅 | 事件标题、时间、提醒状态 | 点击编辑、标记完成、删除 |
| **备忘** | 左侧黄色竖线 + 📝 | 备忘内容、关联提醒 | 点击编辑、归档、删除 |
| **流水账** | 左侧绿色竖线 + 💬 | 原始内容、标签 | 点击编辑标签、删除 |
| **打卡** | 左侧紫色竖线 + ✅ | 打卡项名称、连续天数 | 点击查看统计 |
| **知识** | 左侧橙色竖线 + 💡 | 知识摘要、标签 | 点击展开、编辑、关联 |

卡片默认以紧凑模式显示（1-2 行），点击展开查看完整内容和操作按钮。左滑卡片可快速删除或归档。

**日期分隔线：** 当时间线跨越不同日期时，显示日期分隔线（如"今天"、"昨天"、"3月12日 周三"）。

### 3.3 顶栏

顶栏保持极简：

- **左侧**：JifyLife logo/文字
- **右侧**：用户头像（点击进入个人设置页）、更多菜单（···）

更多菜单展开后提供：
- **日记回顾**：查看 AI 生成的每日/每周日记
- **打卡面板**：查看所有打卡项和统计
- **知识库**：浏览知识卡片
- **搜索**：全文搜索所有记录
- **设置**：AI 模式、主题、通知偏好等

这些功能以**底部弹出抽屉（Bottom Sheet）**或**全屏覆盖层（Overlay）**的形式展现，而不是跳转到新页面。用户完成操作后，下滑或点击关闭即可回到主时间线。

## 4. 语音输入详细设计

### 4.1 技术方案

语音识别采用 **Web Speech API**（浏览器原生）作为主方案，配合后端 **Whisper API** 作为降级方案：

| 方案 | 优势 | 限制 | 适用场景 |
|------|------|------|---------|
| Web Speech API | 零延迟、免费、实时转写 | 浏览器兼容性、需要在线 | 主方案，支持的浏览器优先使用 |
| Whisper API（后端） | 高准确率、支持多语言 | 需要录音上传、有延迟 | 降级方案，Web Speech 不可用时使用 |

### 4.2 语音录制界面

点击麦克风后，底部输入栏扩展为语音录制面板：

```
┌───────────────────────────────────┐
│                                   │
│     ~~~~~ 波形动画 ~~~~~          │
│                                   │
│   "下午三点开会，提醒我..."       │  ← 实时转写文字
│                                   │
│        ⏹  00:05                  │  ← 停止按钮 + 录制时长
│                                   │
│   [取消]              [完成发送]  │
│                                   │
└───────────────────────────────────┘
```

**交互细节：**

- 录制开始时有轻微震动反馈（Haptic Feedback）
- 波形动画实时反映音量大小，给予用户"正在听"的信心
- 实时转写文字随说随显，用户可以看到识别结果
- 停顿 1.5 秒自动结束录制（可在设置中调整或关闭）
- 结束后转写文字进入可编辑状态，用户可修改后发送
- 支持长按麦克风按钮的"按住说话"模式（可选，在设置中开启）

### 4.3 语音指令识别

AI 不仅转写语音内容，还能识别特殊指令模式：

| 语音输入示例 | AI 解析行为 |
|-------------|------------|
| "下午三点开会" | → 创建日程：15:00 开会 |
| "提醒我明天带伞" | → 创建备忘 + 设置明天提醒 |
| "今天跑了五公里" | → 流水账 + 识别"跑步"打卡项 |
| "刚学到一个知识：原来..." | → 创建知识卡片 |
| "打卡喝水" | → 记录"喝水"打卡 |
| "看看我这周的日记" | → 打开日记回顾面板 |

## 6. 视觉设计规范

### 6.1 色彩体系

延续 PRD 中的柔和渐变风格，以**中性温暖色调**为基底，配合功能色区分卡片类型：

| 用途 | 色值 | 说明 |
|------|------|------|
| 品牌主色 | `#6366F1` (Indigo-500) | 渐变起始色，用于品牌标识和主要操作 |
| 品牌辅色 | `#8B5CF6` (Violet-500) | 渐变终止色，配合主色使用 |
| 背景色 | `#FAFAF9` (Stone-50) | 主背景，微暖白 |
| 卡片背景 | `#FFFFFF` | 纯白卡片，配合阴影层次 |
| 文字主色 | `#1C1917` (Stone-900) | 正文文字 |
| 文字辅色 | `#78716C` (Stone-400) | 次要文字、时间戳 |
| 日程色 | `#3B82F6` (Blue-500) | 日程卡片标识色 |
| 备忘色 | `#F59E0B` (Amber-500) | 备忘卡片标识色 |
| 流水账色 | `#10B981` (Emerald-500) | 流水账卡片标识色 |
| 打卡色 | `#8B5CF6` (Violet-500) | 打卡卡片标识色 |
| 知识色 | `#F97316` (Orange-500) | 知识卡片标识色 |

### 6.2 字体与排版

```
标题（顶栏品牌）：  20px / 600 weight / Stone-900
卡片标题：         16px / 500 weight / Stone-900
卡片正文：         14px / 400 weight / Stone-700
辅助文字（时间）：  12px / 400 weight / Stone-400
输入框 placeholder：14px / 400 weight / Stone-300
```

### 6.3 间距与圆角

- **卡片圆角**：16px
- **输入框圆角**：24px（胶囊形状，更友好）
- **按钮圆角**：12px
- **卡片间距**：12px
- **卡片内间距**：16px
- **屏幕水平边距**：16px（移动端）/ 24px（桌面端）

### 6.4 动效规范

| 动效场景 | 持续时间 | 缓动函数 | 说明 |
|---------|---------|---------|------|
| 新卡片入场 | 300ms | ease-out | 从输入栏位置上浮入场，带淡入 |
| 卡片展开/折叠 | 200ms | ease-in-out | 高度过渡 + 内容淡入 |
| Bottom Sheet 弹出 | 350ms | spring(1, 80, 10) | 弹性滑入，有回弹感 |
| 语音波形 | 持续 | linear | 实时跟随音量 |
| AI 处理中 | 持续 | linear | 三点跳动 loading 动画 |
| 卡片左滑删除 | 200ms | ease-out | 跟随手指，释放后滑出 |

## 7. 响应式适配

### 7.1 移动端（< 768px）

这是**主要设计目标**。所有上述设计以移动端为准。

- 输入栏固定在底部，位于虚拟键盘上方
- 时间线占满整个屏幕宽度
- 辅助面板以 Bottom Sheet 形式呈现
- 手势操作：下拉刷新、左滑删除、上拉加载更多

### 7.2 桌面端（>= 1024px）

桌面端在保持单窗口理念的同时，利用更大的屏幕空间：

```
┌──────────────────────────────────────────────────────┐
│  JifyLife                           [搜索]  [头像]   │
│──────────────────────────────────────────────────────│
│          │                          │                │
│  快捷面板 │      时间线卡片流        │   侧边预览    │
│          │                          │                │
│  📅 今日  │  同移动端的时间线        │  点击卡片后    │
│  ✅ 打卡  │  但卡片最大宽度 640px    │  在此展示详情  │
│  📖 日记  │  居中显示               │  和编辑操作    │
│  💡 知识  │                          │                │
│  ⚙ 设置  │  ┌─────────────────┐     │                │
│          │  │ 说点什么...  🎤 ➤│    │                │
│          │  └─────────────────┘     │                │
│          │                          │                │
└──────────────────────────────────────────────────────┘
```

桌面端的输入栏依然在时间线底部，但时间线本身居中显示，左侧增加一个快捷面板作为辅助导航，右侧在点击卡片时展示详情。这样桌面端利用了宽屏空间，但核心交互模式与移动端完全一致。

## 8. 关键用户旅程

### 8.1 首次使用

1. 打开应用 → 简洁的欢迎页，一句话介绍 + "开始记录"按钮
2. 进入主界面 → 空状态，时间线区域显示引导文字："试着说一句话，或者打几个字"
3. 用户第一次输入 → AI 处理后，卡片出现，同时显示轻提示："我帮你分类好了，点击卡片可以修改"

### 8.2 日常使用（典型一天）

```
早上 8:00  打开应用，语音输入"今天上午要交报告"
           → AI 创建日程：上午交报告

中午 12:30 语音输入"中午吃了牛肉面，还不错"
           → AI 归类为流水账，标签：餐饮

下午 15:00 语音输入"刚跑完步，今天跑了 3 公里"
           → AI 归类为流水账 + 识别打卡项"跑步"

晚上 21:00 打开应用，点菜单"日记回顾"
           → 查看 AI 根据今天 3 条记录生成的日记
           → 可编辑后保存

睡前       收到明天"交报告"的提醒推送（PWA 通知）
```

## 9. 技术实现要点

### 9.1 语音输入实现

```typescript
// 语音输入 Hook 的核心接口
interface UseVoiceInput {
  // 状态
  isRecording: boolean;
  isProcessing: boolean;
  transcript: string;        // 实时转写结果
  finalTranscript: string;   // 最终转写结果
  audioLevel: number;        // 音量级别 0-1，驱动波形动画

  // 操作
  startRecording: () => void;
  stopRecording: () => void;
  cancelRecording: () => void;

  // 配置
  autoStop: boolean;         // 是否自动停止
  autoStopDelay: number;     // 自动停止延迟（ms）
  preferredEngine: 'webSpeech' | 'whisper';
}
```

### 9.2 AI 意图识别

用户输入经过后端 AI 服务进行意图识别和内容解析：

```typescript
interface ContentAnalysis {
  type: 'schedule' | 'memo' | 'flow' | 'checkin' | 'knowledge' | 'command';
  confidence: number;          // 置信度 0-1
  parsedData: {
    title?: string;            // 提取的标题/摘要
    datetime?: string;         // 解析出的时间
    reminder?: string;         // 提醒时间
    tags?: string[];           // 自动标签
    checkinItem?: string;      // 打卡项名称
    command?: string;          // 指令名称
  };
  originalContent: string;     // 原始输入
}
```

## 10. 与原 PRD 的变更对照

本设计方案对原 PRD 中的界面部分做了重大简化。功能范围不变，但交互方式从多页面多入口收敛为单窗口单入口。

| 维度 | 原 PRD 设计 | 新设计方案 |
|------|------------|-----------|
| **导航模式** | 多页面，底部 Tab 切换 | 单窗口，无导航 Tab |
| **输入方式** | 进入对应页面后创建 | 统一输入框，AI 自动分类 |
| **语音输入** | 未提及 | 作为主要输入方式 |
| **内容展示** | 分模块页面（日程页、备忘页等） | 统一时间线，按类型标记 |
| **辅助功能** | 独立页面 | Bottom Sheet / Overlay |
| **页面数量** | 7+ 个独立页面 | 1 个主页面 + 2 个覆盖层 |
| **交互复杂度** | 中等，需要学习导航 | 极低，说话/打字即可 |
| **核心功能** | 日程、备忘、日记、打卡、知识 | 完全保留，交互方式变更 |
